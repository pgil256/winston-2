# Dressup Wardrobe Upgrade Design

## Summary

Upgrade Winston 2.0's dress-up feature from four hardcoded dropdowns into a playful, data-driven wardrobe system. The app will keep the current static `public/models/ferret.glb` as the base model and add an expanded accessory pipeline that supports both imported `.glb` accessories and existing/new procedural Three.js accessories.

The wardrobe prioritizes costume variety: wizard, pirate, royal, space, party, cozy, and classic pet-fashion looks. Licensing is CC0-first, with CC-BY allowed only for high-value assets that include source and attribution metadata before they ship.

## Approved Direction

- Use the current static ferret GLB; do not replace it with a rigged/posable ferret for this project.
- Build a hybrid wardrobe pipeline: procedural accessories plus imported GLB accessories.
- Expand from four slots to six: `head`, `face`, `neck`, `body`, `back`, and `feet`.
- Replace the main dress-up experience with a compact wardrobe UI; Leva can remain only for development/debug controls if useful.
- Preserve existing saved looks through a versioned migration path.

## Current Project Context

The repo is a Vite + React 18 + TypeScript app using `three`, `@react-three/fiber`, `@react-three/drei`, `leva`, and `zustand`.

The original `SPEC.md` describes a procedural rigged ferret, but the current `README.md` and implementation now use a static Poly by Google ferret GLB. Accessory rendering currently flows through:

- `src/ferret/FerretModel.tsx`: loads `/models/ferret.glb`.
- `src/ferret/FerretRig.tsx`: places static accessory groups at measured offsets around the GLB.
- `src/accessories/AccessoryAt.tsx`: chooses a procedural accessory component by slot.
- `src/accessories/types.ts`: defines the old four-slot accessory shape.
- `src/ui/ControlPanel.tsx`: exposes Leva dropdowns for the old slots.
- `src/storage/looks.ts`: saves and loads pose, accessories, and environment.

There are existing uncommitted local edits in dress-up-related files. Implementation must preserve and review those edits instead of reverting them.

## Measured Ferret GLB Data

Local inspection of `public/models/ferret.glb` with `@gltf-transform/cli inspect` and a direct vertex-buffer scan found:

- Asset format: glTF 2.0 binary GLB generated by `obj2gltf`.
- Scene/root name: `Ferret_Mesh`.
- Meshes: 1.
- Materials: 1.
- Textures: 1 embedded 2048 x 2048 PNG base-color texture.
- Skins: 0.
- Animations: 0.
- Position vertices: 1,618 uploaded vertices.
- Bounding box in model-local coordinates:
  - min: `[-0.5173, -0.01863, -2.59349]`
  - max: `[0.5173, 2.11011, 1.82064]`
  - dimensions: `[1.0346, 2.12874, 4.41413]`
- Current scale-to-length target: `0.135927`, producing an approximate scene footprint of `0.1406w x 0.2894h x 0.6l`.

Measured candidate attachment regions after current scene scaling:

- `headCrown`: local near `[0, 2.1008, 1.317]`, scene near `[0, 0.2881, 0.2315]`.
- `chestFront`: local region around `y=1.08..1.28`, `z=1.36..1.39`, scene around `y=0.1496..0.1762`, `z=0.2378..0.2416`.
- `upperBack`: local region around `y=1.4075..1.5576`, `z=0.0021..0.7289`, scene around `y=0.1939..0.2143`, `z=0.0528..0.1516`.
- `frontPaws`: scene region around `x=+/-0.0557`, `y=0.0024..0.0212`, `z=0.1489..0.2057`.
- `rearFeet`: scene region around `x=+/-0.0666`, `y=0.0022..0.0237`, `z=-0.0888..-0.0038`.

Because this GLB has no skeleton or animations, accessories cannot follow bones or deform with poses. This project should solve fit through measured static anchors, per-item transforms, preview verification, and save/load state.

## External Asset Sources

Candidate sources and rules:

- Kenney: use as a primary CC0 source. Kenney's official 3D import guide says its game assets distribute glTF as GLB and recommends GLB in several engines. Source: [Kenney importing 3D models](https://kenney.nl/knowledge-base/game-assets-3d/importing-3d-models-into-game-engines).
- Quaternius: use as a primary CC0 source for stylized low-poly modular costume parts. The Modular Character Outfits - Fantasy page lists glTF format and CC0 license. Source: [Quaternius Modular Character Outfits - Fantasy](https://quaternius.com/packs/modularcharacteroutfitsfantasy.html).
- Poly / Poly Pizza: allow only as CC-BY when the asset is clearly worth the extra attribution. Google's Poly help states remixable Poly assets are CC-BY 3.0 and require attribution. Source: [Google Poly Creative Commons help](https://support.google.com/poly/answer/7418679?hl=en). The existing ferret source is [Poly Pizza Ferret](https://poly.pizza/m/4I1SBFHWuSo).
- glTF/GLB compatibility: GLB is the binary glTF container and is appropriate for web delivery. Sources: [Khronos glTF overview](https://www.khronos.org/gltf/), [Khronos glTF 2.0 specification](https://registry.khronos.org/glTF/specs/2.0/glTF-2.0.html), and [Three.js GLTFLoader docs](https://threejs.org/docs/pages/GLTFLoader.html).

## Goals

- Make dress-up immediately more fun by shipping a visibly larger wardrobe.
- Support imported GLB accessories and procedural accessories through one registry.
- Make every wardrobe item self-documenting: slot, transform, source, license, tags, and renderer type.
- Add player-facing wardrobe controls that work faster than Leva dropdowns.
- Keep the app fully client-side and compatible with the existing Vite/React/Three stack.
- Keep old saved looks usable through migration.

## Non-Goals

- Do not rig, skin, or replace the ferret model in this project.
- Do not add cloth simulation or mesh deformation.
- Do not add user-uploaded assets.
- Do not add backend storage.
- Do not build screenshot/export workflows.

## Architecture

The wardrobe pipeline has three layers.

1. Asset inputs

   Imported `.glb` files live under `public/models/accessories/`. Procedural items continue to live as React/Three components. Every item, regardless of source, has an entry in the wardrobe registry.

2. Registry and anchors

   A typed wardrobe registry describes available items. A separate static-anchor module exposes measured points around the ferret GLB. Each item chooses an anchor and applies its own position, rotation, and scale adjustment.

3. Runtime rendering and UI

   A single wardrobe item renderer reads the active wardrobe state. If the selected item is procedural, it renders the mapped component. If it is GLB-backed, it loads the GLB with drei/Three GLTF loading, clones it safely, applies shadows/material handling, and places it under the anchor transform.

## Data Model

Add `src/wardrobe/types.ts` with types equivalent to:

```ts
export type WardrobeSlot = 'head' | 'face' | 'neck' | 'body' | 'back' | 'feet';

export type WardrobeAnchor =
  | 'headCrown'
  | 'faceBridge'
  | 'chestFront'
  | 'upperBack'
  | 'waist'
  | 'frontPawL'
  | 'frontPawR'
  | 'rearFootL'
  | 'rearFootR';

export type LicenseKind = 'CC0' | 'CC-BY-3.0' | 'CC-BY-4.0' | 'Custom';

export interface WardrobeSource {
  readonly name: string;
  readonly url: string;
  readonly license: LicenseKind;
  readonly attribution?: string;
}

export interface WardrobeTransform {
  readonly anchor: WardrobeAnchor;
  readonly position: readonly [number, number, number];
  readonly rotation: readonly [number, number, number];
  readonly scale: number | readonly [number, number, number];
}

export interface BaseWardrobeItem {
  readonly id: string;
  readonly label: string;
  readonly slot: WardrobeSlot;
  readonly source: WardrobeSource;
  readonly transform: WardrobeTransform;
  readonly tags: readonly string[];
}

export interface GlbWardrobeItem extends BaseWardrobeItem {
  readonly kind: 'glb';
  readonly assetPath: `/models/accessories/${string}.glb`;
}

export interface ProceduralWardrobeItem extends BaseWardrobeItem {
  readonly kind: 'procedural';
  readonly component: string;
}

export type WardrobeItem = GlbWardrobeItem | ProceduralWardrobeItem;

export type WardrobeSelection = Record<WardrobeSlot, string | 'none'>;
```

The exact implementation may add helper types for multi-anchor items, especially `feet`, but the public selection state remains one active item per slot.

## Slots And Anchors

Slots:

- `head`: hats, crowns, helmets, party pieces.
- `face`: glasses, visors, masks, monocles.
- `neck`: collar, scarf, bowtie, bandana.
- `body`: sweater, tutu, armor-like chest pieces.
- `back`: cape, wings, backpack, jetpack.
- `feet`: socks, boots, slippers; rendered across all four foot anchors.

Static anchors:

- `headCrown`: top of skull.
- `faceBridge`: front of face/snout bridge, derived from the same local scan as `headCrown` and chest-front profiles.
- `chestFront`: front upper chest for bowties, collars, and bandanas.
- `upperBack`: shoulders/back for capes, backpacks, and wings.
- `waist`: mid-body rear/waist for tutu and belt-like items.
- `frontPawL`, `frontPawR`, `rearFootL`, `rearFootR`: feet anchors.

Anchor data should live in `src/wardrobe/anchors.ts` and expose both raw measured points and scene-space helper transforms. Keep the measured local coordinates in code comments so future tuning can be traced.

## Initial Wardrobe

Ship a first upgraded wardrobe with enough variety to prove the system:

- Head: wizard hat, pirate hat, crown, space helmet, party hat, top hat.
- Face: sunglasses, star glasses, space visor, pirate eyepatch.
- Neck: bowtie, collar, scarf, bandana.
- Body: sweater, tutu, armor/chest piece.
- Back: cape, wings, backpack or jetpack.
- Feet: boots, socks, slippers.

Not every item must be GLB. Exact-fit items such as scarf/collar/feet can remain procedural when that gives better results. Imported GLB items should be used where they add visible richness without a long custom modeling effort.

## Wardrobe UI

The main user-facing UI should be a compact overlay instead of four Leva dropdowns.

Required controls:

- Slot tabs for `Head`, `Face`, `Neck`, `Body`, `Back`, and `Feet`.
- Item tiles for the active slot, with label and short tag/category text.
- Clear current slot.
- Reset outfit.
- Randomize outfit.
- Save/load named look.

The UI should allow a user to create a funny full outfit in under 10 seconds. Leva can stay for debug-only environment/dev controls if it remains useful, but the wardrobe flow should not depend on Leva.

## Save/Load Migration

Introduce a versioned look schema:

- Version 1: existing shape with `accessories: { head, neck, body, feet }`.
- Version 2: new shape with `wardrobe: WardrobeSelection`, plus environment and any existing pose fields retained for backward compatibility.

When loading a version 1 look:

- Map `head` to `wardrobe.head`.
- Map `neck` to `wardrobe.neck`.
- Map old `body` values:
  - `cape` maps to `wardrobe.back`.
  - `sweater` and `tutu` map to `wardrobe.body`.
- Map `feet` to `wardrobe.feet`.
- Fill all missing new slots with `'none'`.

Saving after migration writes version 2.

## Asset Intake Workflow

Every imported GLB should go through a repeatable intake checklist:

1. Confirm license and source URL.
2. Place raw candidate outside shipping path until accepted.
3. Inspect with `@gltf-transform/cli inspect`.
4. Record bbox, mesh count, texture count, file size, and animations/skins if present.
5. Reject assets that are too heavy, visually inconsistent, or impossible to fit to a static anchor.
6. Optimize accepted assets with glTF Transform when useful.
7. Place accepted GLBs in `public/models/accessories/`.
8. Add registry metadata and credits in the same change.
9. Verify in browser on the ferret from front, side, and three-quarter camera angles.

## Error Handling

- If a GLB accessory fails to load, show no item for that slot and log a concise console warning with the item id and asset path.
- Keep the rest of the scene interactive when one accessory fails.
- Registry validation tests should catch missing files and invalid metadata before runtime.
- For old saved looks that reference unknown item ids, fall back to `'none'` for that slot.
- For malformed saved-look JSON, keep the existing storage behavior: ignore the bad entry rather than crashing.

## Testing And Verification

Unit tests:

- Registry item ids are unique.
- Every item references an existing slot.
- Every item references an existing anchor.
- Every item has source name, source URL, and license.
- CC-BY items include attribution text.
- Every GLB item references an existing file under `public/models/accessories/`.
- Version 1 looks migrate into the version 2 wardrobe selection correctly.
- Store actions update one wardrobe slot at a time.
- Reset clears all wardrobe slots.
- Randomize returns valid item ids or `'none'`.

Build verification:

- Run `npm test`.
- Run `npm run build`.

Browser verification:

- Start the Vite dev server.
- Open the app in the browser.
- Verify at least five complete outfits: wizard, pirate, royal, space, and cozy/pet fashion.
- Check front, side, and three-quarter views for floating, clipping, and scale problems.
- Save and load one version 2 outfit.
- Load a version 1 saved look fixture and verify it maps into the new slots.
